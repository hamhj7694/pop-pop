# TOKTOK 개발 계획

이 문서는 `PRD.md`와 `docs/*-rules.md`를 기준으로 한 개발 로드맵이다. 실제 진행 상태 관리는 루트의 `TODO.md`에서 하고, 완료된 계획은 `docs/completed-plans.md`에 기록한다.

## 계획 원칙

- 개발 순서는 `손맛 > 감각적 만족 > 랜덤 보상 > 수집 욕구 > 운영 기능` 순서를 따른다.
- 첫 화면은 항상 플레이 가능한 화면이어야 한다.
- 보상, 안내, 설정, 도감은 터뜨리는 흐름을 막지 않는 방식으로 구현한다.
- 모바일 터치 드래그와 한 손 조작을 핵심 검증 기준으로 둔다.
- MVP 제외 항목인 강제 광고, 결제, 랭킹, 친구, 실시간 커뮤니티, 푸시, AI 추천은 만들지 않는다.

## Phase 0. 프로젝트 기반 구축

목표: PRD 기준 기술 스택으로 개발 가능한 최소 프로젝트를 만든다.

주요 작업:

- Vite + React + TypeScript 프로젝트 초기화
- Tailwind CSS 설정
- Zustand, Framer Motion, Howler.js 설치
- 기본 폴더 구조 구성
- 도메인 타입과 상수 초안 작성
- 기본 라우팅 또는 화면 전환 구조 결정
- lint, format, build 스크립트 확인

완료 기준:

- 로컬 개발 서버가 실행된다.
- 첫 화면에서 임시 플레이 화면이 보인다.
- TypeScript build가 통과한다.

## Phase 1. 핵심 손맛 구현

목표: 사용자가 접속 직후 바로 버블을 터뜨릴 수 있게 한다.

주요 작업:

- 반응형 버블 배치 알고리즘 구현
- 버블 상태 모델 구현
- 클릭/터치 터뜨리기 구현
- 마우스 드래그와 모바일 터치 드래그 연속 터뜨리기 구현
- 중복 터뜨림 방지
- 기본 눌림/터짐 애니메이션 구현
- 기본 파티클 구현
- 터진 버블 자동 재생성 구현

완료 기준:

- 데스크톱과 모바일 크기에서 버블이 화면에 자연스럽게 배치된다.
- 클릭, 터치, 드래그로 버블이 즉시 터진다.
- 이미 터진 버블은 중복 처리되지 않고, 잠시 뒤 같은 자리에서 다시 터뜨릴 수 있게 재생성된다.
- 빠르게 터뜨려도 눈에 띄는 입력 지연이 없다.

## Phase 2. 감각 피드백과 설정

목표: 사운드, 진동, 이펙트가 손맛을 강화하고 사용자가 제어할 수 있게 한다.

주요 작업:

- Howler.js 기반 사운드 래퍼 구현
- 기본 터짐 사운드 랜덤 재생
- 연속 입력 중 사운드 중복 제한
- Vibration API 래퍼 구현
- 일반/희귀/초희귀/피버 진동 패턴 준비
- 설정 상태 구현
- 사운드 ON/OFF, 볼륨, 진동 ON/OFF, 이펙트 강도 UI 구현
- 설정 로컬 저장

완료 기준:

- 사운드와 진동은 기본 ON이다.
- 설정에서 사운드와 진동을 독립적으로 끌 수 있다.
- 미지원 진동 환경에서 오류가 발생하지 않는다.
- 사운드 OFF 또는 진동 OFF 상태에서도 플레이가 자연스럽다.

## Phase 3. 보상 루프 구현

목표: 터뜨리는 흐름을 막지 않는 랜덤 보상과 임시 수집 경험을 만든다.

주요 작업:

- 보상 타입, 희귀도, 샘플 보상 데이터 작성
- 랜덤 보상 확률 엔진 구현
- 터진 위치 기준 보상 드랍 연출 구현
- 화면 하단 쌓임 연출 구현
- `오늘 주운 것들 N개` 하단 고정 버튼 구현
- 오늘 주운 것들 오버레이 구현
- 희귀도 필터와 새 보상 강조 구현
- 보상 상세 보기 구현

완료 기준:

- 보상은 팝업이 아니라 드랍/쌓임 방식으로 보인다.
- 오늘 주운 것들 카운트가 획득 수와 일치한다.
- 오버레이는 사용자가 직접 열 때만 표시된다.
- 보상 확인 중에도 다시 플레이로 쉽게 돌아갈 수 있다.

## Phase 4. 도감과 장기 수집

목표: 보상을 장기 수집 대상으로 저장하고 재방문 이유를 만든다.

주요 작업:

- 도감 데이터 모델 구현
- 획득/미획득 상태 표시
- 전체 수집률과 카테고리별 수집률 구현
- 일반/희귀/초희귀/피버 전용 탭 구현
- 최근 획득 항목 표시
- 게스트 상태의 도감 저장 유도 정책 정리
- 로컬 저장 기반 MVP 도감 구현

완료 기준:

- 획득한 보상은 도감에 저장된다.
- 미획득 콘텐츠는 `???`로 표시된다.
- 희귀도는 색상 외 라벨이나 아이콘으로도 구분된다.
- 도감은 플레이 흐름으로 쉽게 복귀할 수 있다.

## Phase 5. 콤보와 피버타임

목표: 연속 터뜨리기의 몰입감을 축하형 시스템으로 강화한다.

주요 작업:

- 콤보 카운트와 종료 타이머 구현
- 콤보 단계 감지 구현
- 콤보 보상 modifier 구현
- 콤보 종료 결과 표시 구현
- 피버타임 발동 확률 구현
- 피버타임 남은 시간과 상태 구현
- 피버타임 배경/버블/파티클/사운드/진동 연출 구현
- 피버 전용 보상 카테고리 구현

완료 기준:

- 콤보는 입력이 이어지는 동안 자연스럽게 증가한다.
- 콤보 종료는 실패가 아니라 기록 달성으로 표현된다.
- 피버타임은 즉시 알아차릴 수 있지만 피로하지 않다.
- 피버타임 중 보상 확률과 연출이 강화된다.

## Phase 6. Supabase 연동과 데이터 영속화

목표: MVP 데이터를 Supabase에 저장하고 계정 기반 수집을 가능하게 한다.

주요 작업:

- Supabase 프로젝트 환경 변수 구조 정리
- DB schema 초안 작성
- `rewards`, `user_rewards`, `user_settings`, `play_sessions` 구현
- Auth 연동
- 게스트 데이터와 로그인 데이터 병합 정책 구현
- Storage 버킷 설계
- RLS 정책 작성
- 기본 seed 데이터 작성

완료 기준:

- 로그인 사용자의 보상과 설정이 서버에 저장된다.
- RLS 정책으로 사용자 데이터 접근이 제한된다.
- 비활성 보상은 신규 드랍 대상에서 제외된다.
- 기존 획득 기록은 보상 풀 변경으로 깨지지 않는다.

## Phase 7. 사용자 제보와 관리자 운영

목표: 사용자 제출 콘텐츠를 검수 후 보상 풀에 넣는 운영 흐름을 만든다.

주요 작업:

- 콘텐츠 제보 화면 구현
- 텍스트/이미지/GIF 제출 지원
- 제출 동의와 검수 안내 구현
- `submissions` 상태 흐름 구현
- 관리자 승인 대기 목록 구현
- 관리자 상세 보기, 승인, 거절, 거절 사유 구현
- 승인 콘텐츠의 보상 풀 등록 구현
- 운영자 직접 보상 등록과 비활성화 구현

완료 기준:

- 사용자 제출 콘텐츠는 승인 전 공개되지 않는다.
- 관리자는 승인/거절/비활성화를 처리할 수 있다.
- 승인된 콘텐츠만 보상 풀에 들어간다.
- 관리자 권한은 클라이언트 표시만으로 의존하지 않는다.

## Phase 8. 품질 안정화와 MVP 릴리스 준비

목표: 성능, 접근성, 모바일 품질, 분석 이벤트를 릴리스 가능한 수준으로 정리한다.

주요 작업:

- 모바일 터치 드래그 집중 테스트
- 60FPS에 가까운 인터랙션 유지 확인
- 파티클과 사운드 중복 최적화
- 키보드와 스크린 리더 접근성 점검
- PostHog 이벤트 연결
- 고빈도 이벤트 샘플링 또는 집계 정책 적용
- 빌드, 린트, 주요 플로우 테스트 정리
- Vercel 배포 준비

완료 기준:

- 초기 로딩 2초 이하를 목표로 최적화가 진행된다.
- 주요 플레이 플로우가 모바일과 데스크톱에서 동작한다.
- 분석 이벤트가 중복 과다 전송되지 않는다.
- MVP 제외 기능이 섞여 들어가지 않았다.

## Phase 9. 테마와 콘텐츠 확장

목표: 기본 플레이 감각을 해치지 않으면서 뽁뽁이 테마, 상자 줍기, 보상 콘텐츠의 다양성을 확장한다.

주요 작업:

- 바닥에 쌓인 상자를 드래그로도 주울 수 있게 구현
- 상자 줍기와 뽁뽁이 터뜨리기 입력 충돌 방지
- 뽁뽁이 테마 데이터 모델과 선택 UI 구현
- 선택 테마의 로컬 저장과 화면 스타일 반영
- 테마별 버블, 배경, 파티클, 상자 스타일 정리
- 문구, 아이템, 이미지/사진, 경품/쿠폰 보상 풀 확장
- 상자 개봉 결과와 보상 타입별 연출을 데이터 기반으로 정리
- 확장 후 모바일 성능과 시각 명확성 점검

완료 기준:

- 사용자는 상자를 하나씩 누르지 않아도 드래그로 자연스럽게 주울 수 있다.
- 테마 변경 후에도 뽁뽁이 식별성과 터뜨리는 손맛이 유지된다.
- 보상 콘텐츠가 타입별로 구분되고, 수집함에는 중복 없이 저장된다.
- 피버타임, 파티클, 보상 연출이 선택 테마와 충돌하지 않는다.
- 보상 확장이 핵심 플레이 흐름을 막지 않는다.

2. 오늘 주운 것들 오버레이

4. 도감 화면
5. 테마 선택 화면
6. 콘텐츠 제보 화면
7. 관리자 화면

## 주요 리스크

- 버블 수와 파티클이 많아질 때 렌더링 성능이 떨어질 수 있다.
- 연속 터뜨리기 중 사운드가 과도하게 겹칠 수 있다.
- 보상이 너무 자주 또는 너무 크게 보이면 핵심 흐름을 방해할 수 있다.
- 테마 색상이 너무 약하면 버블과 배경 구분이 어려워질 수 있다.
- 상자 드래그 줍기가 버블 드래그 입력을 방해할 수 있다.
- 모바일 진동은 브라우저와 기기 지원 차이가 크다.
- 사용자 제보 콘텐츠는 검수와 권한 정책이 부실하면 운영 리스크가 커진다.
